# Forge V1.2 — Plan Mode & Task Tracking

Draft 2026-07-10. Extends V1.1 (spec: `2026-07-10-forge-v1.1-workspace-design.md`). Two complementary features: **plan mode** (the agent explores read-only, proposes a plan, the user approves before execution) and **todos** (a live task checklist the agent maintains during a run). Approving a plan flows naturally into a todo list that tracks execution.

## Goals and non-goals

**Goals**
1. **Plan mode**: a per-session mode (`act | plan`). In plan mode the agent can only use read-only tools; it finishes by calling `propose_plan`, which gates the run on user approval (like an approval gate, with its own richer card). Approve → session flips to act mode and the run continues into execution. Request changes → feedback goes back as the tool result and the agent revises, still in plan mode.
2. **Todos**: an `update_todos` tool (full-snapshot semantics, like Claude Code's TodoWrite) the agent calls to maintain `{text, status: pending|in_progress|completed}` items. Rendered as a live checklist strip pinned above the composer.
3. Both durable and replayable: mode, the proposed/approved plan, and every todo snapshot are events in the session log; the UI stays a pure projection.

**Non-goals:** plans as standalone project artifacts (files, CRUD, cross-session plans); nested/hierarchical todos; auto-generated todos without the model's cooperation; per-plan-step approval; keyboard mode cycling (Shift+Tab); starting sessions in plan mode from dialogs (palette flip covers it).

## Engine

### Mode

- `Mode = Literal["act", "plan"]` in `events.py`. `SessionMeta` gains `mode: Mode = "act"`.
- New durable event `ModeChanged(mode)`; applied in `_replay_meta`. `SessionCreated` unchanged (sessions always start in act mode).
- `SessionActor.set_mode(mode)` mirrors `set_autonomy` (meta + emit). Allowed while running: the tool list is recomputed every loop iteration, so a flip takes effect on the next completion; the in-flight guard below covers the current one.
- `POST /api/sessions/{sid}/mode` `{mode}` → 400 on invalid value.

### Plan mode mechanics (actor)

- **Tool filtering:** in `_loop`, when `meta.mode == "plan"` the specs sent to the model are only tools with `read_only = True` plus `propose_plan`. (`bash` is excluded entirely — it can mutate; `spawn_agents` is included but the actor forces `mode="read"` on every task while in plan mode.)
- **Defense in depth:** `_execute_call` refuses non-read-only tools while in plan mode with an error tool result: `"Blocked: session is in plan mode. Only read-only tools are available; finish by calling propose_plan."` (Handles stale in-flight calls and model disobedience without killing the run.)
- **`propose_plan` tool** (`server/forge/tools/plan.py`): params `{plan: string}` (markdown). It is a *gating* tool, executed specially by the actor (like `_gate`), not via `tool.run`:
  1. Emit durable `PlanProposed(call_id, plan)`, set status `attention`.
  2. Await a future resolved by `POST /api/sessions/{sid}/plan/{call_id}` `{decision: "approve" | "revise", feedback?: string}` (reuses the `_approvals` future map — call ids are unique).
  3. Emit durable `PlanResolved(call_id, decision, feedback)`.
  4. **approve** → `set_mode("act")` (emits `ModeChanged`), tool result: `"Plan approved. You are now in act mode — execute the plan. Start by calling update_todos with the plan's steps."`
  5. **revise** → stays in plan mode, tool result: `"User requested changes to the plan: {feedback}"` (`is_error=False` — revision is a normal part of the loop).
- Cancel/restart while awaiting the gate: identical to approval gates — the dangling call is closed by `_close_dangling`, `RunFinished(cancelled|interrupted)` follows. The session stays in plan mode.
- **System prompt:** when `meta.mode == "plan"`, `build_system_prompt` appends a `## Plan mode` section: read-only exploration, produce a concrete reviewable plan, call `propose_plan` exactly once when ready, do not attempt writes.

### Todos

- `Todo(BaseModel)`: `text: str`, `status: Literal["pending", "in_progress", "completed"]` (in `events.py`).
- **`update_todos` tool** (`server/forge/tools/todos.py`): params `{todos: [{text, status}]}` — the full list every call (snapshot semantics; the model rewrites the whole list). `read_only = True` (no approval gate; it touches nothing outside the log). Validation errors (bad status, empty list item) → error tool result.
- The tool returns the parsed snapshot on a new optional `ToolResult.todos: list[Todo] | None = None` field (mirrors the `diff_stats` pattern); when present, the actor emits durable `TodosUpdated(todos)` right after `ToolCallFinished`. The reducer keys off `todos_updated`, never off parsing tool args.
- `update_todos` is available in **both** modes (in plan mode it lets the agent sketch progress on research, and it is read-only anyway).
- **Compaction:** the summarizer prompt gains "…the approved plan if any, and the current todo list with each item's status." so plan/todos survive `context_compacted` in the summary.
- **System prompt:** `## Guidelines` gains one line: use `update_todos` to track multi-step work; keep exactly one item `in_progress`; update immediately as steps complete.

### Events (protocol additions)

Durable, all in the `Event` union and `_replay_meta` where noted:

- `ModeChanged { mode }` — replayed into meta.
- `PlanProposed { call_id, plan }`
- `PlanResolved { call_id, decision: "approve" | "revise", feedback: str = "" }`
- `TodosUpdated { todos: Todo[] }`

`SessionMeta` gains `mode`. No new ephemeral events. Regenerate `web/src/protocol/generated.ts` (`pnpm gen:protocol`); export `Todo` and `Mode`; add the three literals to the contract test.

### Projection (`to_messages`)

`plan_proposed` / `plan_resolved` / `todos_updated` / `mode_changed` produce no messages — the story the model sees is already complete: `assistant_message` carries the `propose_plan`/`update_todos` tool calls, and `tool_call_finished` carries the outcome (approval text, feedback, or "Todos updated").

### REST summary

- `POST /api/sessions/{sid}/mode` `{mode}` → `{}`; 400 invalid.
- `POST /api/sessions/{sid}/plan/{call_id}` `{decision, feedback?}` → `{}`; 400 invalid decision; resolving an unknown/expired call id is a no-op (same contract as approvals).

## Web

### Reducer

`SessionStream` gains `mode: Mode` (seeded from meta, updated by `mode_changed`) and `todos: Todo[]` (replaced wholesale by `todos_updated`). New stream item:

- `{ kind: 'plan'; seq; callId; plan: string; state: 'pending' | 'approved' | 'revising' }`
  - `plan_proposed` pushes it (`pending`).
  - `plan_resolved` with `approve` → `approved` (the card stays in the stream as the plan of record, collapsed).
  - `plan_resolved` with `revise` → `revising` (muted, feedback shown); a later `plan_proposed` pushes a fresh card.
- The `update_todos` tool card renders compactly (like other read-only tools); the checklist itself lives in the strip, not the stream.

### PlanCard (`web/src/components/PlanCard.tsx`)

Stream-width card (2a language: `--bg-raised-1`, hairline, `--r-card`):

- Header: `PLAN` mono label + state badge (`awaiting approval` warn / `approved` accent / `changes requested` muted).
- Body: plan markdown via `react-markdown` (same styling as prose). Approved/revising cards collapse to header + first heading, click to expand.
- Pending actions: **Approve & execute** (accent) and **Request changes** (ghost → reveals inline textarea + Send). Both POST the resolution; the card state updates via the event, not optimistically.

### TodoStrip (`web/src/components/TodoStrip.tsx`)

Pinned between ChatStream and Composer, only when `todos.length > 0`:

- Collapsed (default): one row — progress glyph `◐ 3/7`, the current `in_progress` item's text ellipsized, chevron.
- Expanded: checklist — `✓` muted-strikethrough for completed, `◐` accent for in_progress, `○` for pending. Max-height with scroll.
- Pure projection of `stream.todos`; no interaction beyond expand/collapse (local state).

### Mode surfacing

- Composer pill: `plan · ` prefix joins the existing `{effort} · {model}` when mode is `plan`, plus the composer border tints `--warn-dot`-ish while in plan mode (subtle, like archived's disabled treatment but active).
- Palette: `/mode` command with `act | plan` submenu (mirrors `/effort`). Store action `setMode(mode)` → `api.setMode`.
- Status: awaiting plan approval reuses `attention` (dot + queue pill already handle it).

## Error handling

- Plan resolution race (run cancelled between render and click): the POST is a no-op; the gate card is superseded by the closed-dangling tool result on replay.
- `propose_plan` called in act mode: executes normally as a gate anyway (harmless; the model gets approval text but mode simply stays `act`). No special casing.
- Invalid `update_todos` payload → error tool result; no `TodosUpdated` emitted; the strip keeps the last good snapshot.

- **Web (vitest):** reducer `mode_changed`/`todos_updated`/plan item lifecycle (pending → approved / revising, second proposal); PlanCard actions post the right bodies; TodoStrip render/progress/collapse; `/mode` palette; composer pill with plan prefix.
